# ERP UI/UX 設計規範 (UI/UX Design Guidelines)

為了維持系統整體的視覺一致性與專業感，所有功能模組（設備、硬體、耗材等）的開發必須遵循以下介面設計規則：

## 1. 資料列表設計 (Table List Standardization)
所有資料表格應具備以下特徵：

*   **資訊層次化 (Info Hierarchy)**：
    *   首欄標題：統一為「廠牌 / 型號 / 類型」。
    *   內容顯示：第一行為 **粗體標題 (13px, #1e293b)** 顯示廠牌；第二行為 **灰色小字 (11px, #64748b)** 顯示 `類型 - 型號`。
*   **序號特徵 (SN Styling)**：
    *   序號 (SN) 欄位內容必須使用 **藍色 (#2563eb)**。
    *   字體加粗至 **800**，並採用 **等寬字體 (Monospace)**。
*   **視覺風格 (Visuals)**：
    *   表格標題列 (`<thead>`) 統一背景色為 **#f8fafc**，底線 **2px solid #f1f5f9**。
    *   標題文字必須設為 **極粗體 (Weight: 900)**，顏色使用高對比深色 **#1e293b**。
    *   數值型資料（如：庫存、數量）應置中對齊。
*   **操作選單 (Action Menu)**：
    *   不直接顯示多個按鈕，統一使用 **MoreHorizontal (Lucide Icon)** 下拉式選單。
    *   編輯功能應放在選單首位，刪除功能置於末位並配以紅底警示風格。

### 1.1 保固資訊顯示規範 (Warranty Info Layout)
為了節省表格橫向空間並維持易讀性，所有設備清單的保固與日期資訊應合併為單一欄位「保固資訊 (P/S/W/C)」，並遵循以下佈局：

*   **2x2 矩陣佈局 (Grid Structure)**：
    *   使用 `display: grid` 與 `gridTemplateColumns: '1fr 1fr'` 實作。
    *   **第一列**：顯示 **P** (裝機日期) 與 **S** (系統日期)。
    *   **第二列**：顯示 **W** (原廠保固) 與 **C** (客戶保固)。
*   **縮寫與色彩辨識 (Labels & Colors)**：
    *   每個數值前必須帶有單字母縮寫與特定的粗體顏色：
        *   **P:** 裝機日期 - **藍色 (#2563eb)**
        *   **S:** 系統日期 - **綠色 (#059669)**
        *   **W:** 原廠保固 - **紅色 (#dc2626)**
        *   **C:** 客戶保固 - **橘色 (#d97706)**
*   **視覺風格**：
    *   字體大小統一為 **10px**。
    *   當資料缺失時，顯示 `--` 以維持對齊。
    *   儲存格最小寬度建議設為 **150px**。

### 1.2 客戶與位置顯示規範 (Customer & Location Layout)
為了讓使用者能快速識別歸屬與地點，所有清單的客戶與位置欄位應遵循以下標準：

*   **標題與圖示 (Icons & Titles)**：
    *   欄位名稱統一簡化為「客戶」與「位置」。
    *   **客戶**：內容前須配有 `<User size={14} color="#64748b" />`。
    *   **位置**：內容前須配有 `<MapPin size={14} color="#64748b" />`。
*   **字體與對齊 (Typography & Alignment)**：
    *   主體內容文字與圖示應水平置中對齊 (`alignItems: 'center'`)，間距設為 `gap: '4px'`。
    *   客戶名稱應使用 **粗體 (700)** 以提升辨識度。
*   **附屬資訊處理 (Secondary Info)**：
    *   若有聯絡人或電話等次要資訊，應放置於主內容下方。
    *   次要資訊需進行縮排 (`paddingLeft: '18px'`)，確保與上方主內容的圖示區域對齊，維持視覺垂直線整齊。

### 1.3 自訂欄位管理規範 (Custom Field Management)
為了確保系統彈性與資料完整性，自訂欄位的建立與刪除必須遵循以下邏輯：

*   **語意化 ID 命名 (Semantic IDs)**：
    *   **核心欄位**：應使用具備語意化且固定的 ID，以維持程式邏輯穩定。
        *   `os`: 作業系統 (對應資料庫原生 `os` 欄位)
        *   `nic`: 韌體版本 (對應資料庫原生 `nic` 欄位)
        *   `user_profile`: 使用者設定檔 (存於 `custom_attributes`)
    *   **動態欄位**：由系統產生時間戳 ID (如 `custom_1714...`)，確保 ID 唯一不重複。
*   **視覺呈現標準 (Styling)**：
    *   **標籤 (Label)**：統一使用灰色 (**#64748b**)，字體不加粗，結尾帶有冒號。
    *   **數值 (Value)**：使用各別定義的色彩 (如 `def.color`)，並設為 **加粗 (fontWeight: 600)**。
*   **徹底刪除機制 (Deep Delete)**：
    *   當在介面上刪除欄位定義時，系統必須連動執行資料庫清理 (`UPDATE assets SET custom_attributes = custom_attributes - 'ID'`)。
    *   確保「幽靈資料」不會殘留在資料庫中占用空間或造成未來 ID 重建時的資料衝突。

### 1.4 硬體與伺服器關聯顯示規範 (Hardware & Server Linkage)
在硬體清單中，為了強化硬體零件與主機之間的關聯性，對應伺服器的顯示需遵循：

*   **垂直排列佈局**：
    *   **第一行**：顯示伺服器序號 (Server SN)，配有 `<Server size={12} />` 圖示，顏色為紫色 (**#6366f1**)。
    *   **第二行**：顯示主機名稱 (HostName)，使用小字 (**11px**) 與灰色 (**#64748b**)，並進行適度縮排 (**16px**) 以維持層次感。
*   **屬性去重原則**：
    *   若 HostName 已在關聯欄位中顯示，則「伺服器屬性」欄位應自動過濾該項目，避免資訊重複堆疊。
所有的「修改詳細資訊」視窗必須遵循以下佈局：

*   **頂部鎖定資料列 (Fixed Header Row)**：
    *   首行標題標記為「廠牌 / 類型 / 型號 (鎖定)」。
    *   下方欄位使用三個並排且 **Disabled (鎖定)** 的輸入框。
    *   鎖定框背景色統一使用 **#f1f5f9**，鼠標為 **not-allowed**。
*   **欄位佈局 (Grid Layout)**：
    *   其餘編輯欄位採用 **1fr 1fr** 的雙欄網格設計。
    *   「規格內容 / 備註」等欄位須佔用完整寬度，並採用 `textarea` (minHeight: 80px)。
*   **底部操作區 (Footer Buttons)**：
    *   上方需有一條 **1px solid #f1f5f9** 的分隔線。
    *   **儲存變更** 按鈕位於左側 (Flex: 1)，背景色為藍色 (#2563eb)。
    *   **取消** 按鈕位於右側，背景色為淺灰色 (#f1f5f9)。

## 3. 互動式統計字卡 (Draggable Stats Cards)
清單上方的統計區塊（Stats Container）應支援自定義佈局：

*   **智慧型網格佈局**：
    *   採用 Grid 佈局：`repeat(6, 1fr)`。
    *   **每一橫排最多顯示 6 張小卡**，超過則自動向下換行（支援無限增加）。
*   **佈局持久化與清理 (Persistence & Purge)**：
    *   排序存儲於 **localStorage**（例如：`hw_list_layout_map`）。
    *   **幽靈字卡清理**：系統必須在渲染前自動清理資料庫中已不存在的遺留 Key (Stale Keys)。
*   **即時自適應收合 (Auto-Collapse)**：
    *   容器必須根據實際存在的卡片位置動態計算高度。
    *   下方若無實體字卡，容器應立即縮回。僅在拖曳期間（Dragging Mode）才動態顯示下一排空格供擴充。

## 4. 佈局與寬度規範 (Layout & Width Standards)
*   **列表欄位寬度**：
    *   **廠牌 / 型號 / 類型**：200px ~ 220px。
    *   **功能按鈕欄**：80px ~ 120px。
    *   **狀態欄位**：約 100px。
*   **視窗尺寸**：
    *   **標準編輯視窗**：600px ~ 680px。
    *   **小型通知視窗**：400px ~ 450px。
*   **通用間距**：卡片內距 **24px** / 表格寬度 `100%` / 內距 **14px**。

## 5. 側邊導航規範 (Sidebar Navigation Standards)
側邊導航欄 (Sidebar) 作為全系統的核心切換點，需確保在功能擴充時維持高度的可用性：

*   **獨立捲動區域 (Independent Scrolling)**：
    *   側邊欄內容導航區必須具備獨立捲軸 (`overflow-y: auto`)，以確保 Logo 與頂部區塊固定不變。
    *   **捲動條樣式**：寬度限制為 **4px**，採用極簡透明設計，避免破壞導航欄的深色底色。
*   **類別聯動收合 (Accordion Interaction)**：
    *   針對多層級類別（如設備、硬體、耗材清單），必須實作「手風琴」邏輯：展開某一類別時，系統必須自動收合其他已展開的類別。
    *   **路徑初始同步 (Initial Path Sync)**：頁面載入或重新整理時，系統必須根據當前 URL 自動展開對應的目錄。
    *   點擊主目錄名稱時，應同步執行「展開選單」與「進入主清單頁面」。
*   **視覺層次與反饋**：
    *   **子選單樣式**：子選單應具備 **20px** 的左側縮排。非作用中項目為半透明白色，目前作用中項目須以 **金黃色 (#FFD700)** 搭配 **加粗 (Weight: 700)** 顯示。
    *   **主選單反饋**：目前作用中的主選單項目，必須在左側顯示 **3px 寬的金黃色垂直邊條**，並搭配背景高亮，提供極佳的導航定位感。
*   **導航狀態同步 (State Synchronization)**：
    *   當透過側邊欄切換不同的子分類（Type / Brand）時，系統必須 **自動清空** 當前頁面的搜尋關鍵字 (`searchTerm`)。
    *   **設計目的**：確保使用者切換至新分類時看到的是完整列表，避免舊的篩選詞與新類別不匹配導致顯示空白。

---
**開發備備註**：新模組開發前，請先參考 `HwList.jsx` 或 `ConsumableList.jsx` 的實作。
## 6. 出貨單建檔規範 (Delivery Note Registration Standards)
針對 ERP 核心的「單據建檔」作業（如 D/N Reg），應遵循以下高密度專業佈局規範：

*   **雙面板非對稱佈局 (Asymmetric Dual-Panel)**：
    *   **左側輸入區 (Input Panel)**：固定寬度 **360px**。包含單據標頭（Header）、序號搜尋（S/N Scan）與品項快選（Quick Pick）。
    *   **右側清單區 (List Panel)**：採用 **Flex: 1** 填滿剩餘空間。以高密度表格（High-density Table）顯示待出貨清單。
*   **高密度視覺控制 (High-Density UI Scaling)**：
    *   **字體級距**：內文與表格統一使用 **0.85rem**；單據標題為 **1.25rem**；標籤（Labels）為 **0.75rem**。
    *   **間距控制**：卡片內距（Padding）縮減至 **12px-16px**，輸入框高度調降以增加垂直資訊量。
*   **序號與關聯硬體自動化 (Automated Serialized Logic)**：
    *   **一鍵導出**：輸入 S/N 透過 `fetchAssetDetailBySN` 查詢時，必須自動帶出廠牌、型號與規格。
    *   **組合包機制 (Bundled Items)**：若設備具備「搭載硬體」（Linked by server_sn），系統必須**自動識別並成組加入**。
    *   **巢狀顯示規範**：主設備顯示於 Main Row，搭載硬體顯示於緊鄰的 Sub Row，並配有縮排圖示（如 ChevronRight）與「搭載硬體」標籤。
*   **耗材快選器 (Consumable Picker)**：
    *   **搜尋觸發機制**：預設狀態隱藏品項，僅在搜尋框有內容時才顯示匹配卡片，以應對未來數百種品項的擴展。
    *   **操作流**：點擊卡片即加入清單，清單內提供數字輸入框（qty-input-small）進行數量微調。
*   **單據草稿持久化 (Draft Persistence)**：
    *   **自動同步**：所有標頭欄位與清單異動必須即時同步至 **localStorage**（例如：`dn_draft_items`）。
    *   **狀態還原**：使用者切換選單或重新載入頁面後，系統必須自動從快取還原草稿，確保操作不中斷。
    *   **提交清理**：僅在執行「送出單據」並確認成功後，才可清除對應的 localStorage 快取。
